# 基于深度学习的图片分类系统

## 摘要

随着人工智能、机器学习和计算机视觉技术的持续发展，图像理解能力已经成为智能系统的重要组成部分。尤其是在自动驾驶、智慧交通、城市治理和工业检测等领域，如何快速、准确地从图片中识别出关键目标，已经成为影响系统可用性和安全性的核心问题。围绕本科毕业设计课题《基于深度学习的图片分类系统》，本文结合自动驾驶道路场景的实际需求，设计并实现了一套面向七类道路目标的图片识别系统。该系统以道路场景图片为输入，能够对图像中的主要交通目标进行分析，并输出分类结果、置信度、目标明细和可视化结果，同时保存历史识别记录，提供后台统计管理功能。

在系统实现方面，本文采用 FastAPI 构建后端服务，使用 Jinja2 与原生 JavaScript 实现前端展示和管理页面，使用 SQLite 保存识别历史数据，在深度学习部分采用 ONNX Runtime 部署 MobileNetV2 预训练模型完成推理能力构建。考虑到毕业设计必须真实部署到资源受限服务器、同时保证上线可运行，本文没有继续使用重量级训练框架，而是采用轻量化整图分类方案：先由 MobileNetV2 输出 ImageNet Top-5 候选类别，再通过类别映射策略转换为行人、自行车、汽车、摩托车、公交车、卡车和交通标志/信号灯七类道路目标，从而在有限资源下兼顾深度学习特征、工程可交付性与答辩展示效果。

为了保证毕业设计项目能够真实交付、稳定运行并完成部署验证，本文对开题报告中的理想技术路线进行了工程化优化：在保留深度学习模型、前台交互、后台管理、数据留痕和部署验证等核心要求的前提下，采用一体化 Python Web 服务方案替代原计划中的多端分离架构，从而降低开发复杂度，缩短实施周期，提高项目完成质量。最终测试结果表明，系统能够稳定完成图片上传、结果展示、历史记录查询、后台统计和服务器部署等功能，具备良好的演示效果和一定的实用价值。本文最后对系统存在的不足进行了分析，并提出了面向专用数据训练、轻量化优化和前后端分离升级的后续改进方向。

**关键词：** 深度学习；图片分类；自动驾驶；ONNX Runtime；FastAPI；系统部署

## 第一章 绪论

### 1.1 研究背景

在信息化和智能化快速发展的今天，图片和视频已经成为人类获取外部信息的重要载体。传统的人工图像判读方式不仅效率低，而且容易受到疲劳、主观经验和环境条件的影响。随着卷积神经网络、迁移学习和大规模预训练模型的发展，深度学习逐渐取代传统手工特征方法，成为图像分类与目标识别的主流技术路线。特别是在自动驾驶和智能交通领域，车辆需要持续理解道路环境中的行人、车辆、交通信号和障碍物等目标，才能做出合理的路径规划和控制决策。因此，构建一套能够对道路场景图片进行识别和分类的系统，不仅具有研究意义，也具有较强的工程实践意义。

自动驾驶道路环境与普通图片分类任务相比，存在更高的复杂性。首先，道路场景通常包含多个目标，且目标尺度差异较大，例如远处的行人和近处的公交车在同一张图中所占比例完全不同。其次，道路场景受到天气、光照、遮挡、运动模糊和拍摄角度的影响较大，不同时间和条件下同一类目标呈现出很大变化。再次，自动驾驶系统对识别结果的稳定性和时效性要求较高，既要求模型具有较好的泛化能力，也要求系统能够在有限算力下完成推理和反馈。基于这些问题，围绕道路场景设计具备工程落地能力的深度学习图片分类系统，具有很强的实践价值。

### 1.2 研究意义

本课题的研究意义主要体现在以下几个方面。

第一，具有明确的应用导向。道路目标识别是自动驾驶感知系统的重要组成部分，是后续决策规划和安全控制的基础。虽然本科阶段的研究不必达到工业级部署标准，但围绕道路目标分类设计一个完整系统，能够让研究更加贴近真实应用场景。

第二，能够锻炼完整的软件工程能力。一个合格的毕业设计不应只停留在模型训练脚本或算法实验层面，而应当尽可能完成需求分析、系统设计、模型接入、前端展示、后台管理、数据存储、部署验证和文档编写等完整链路。通过本课题，可以系统训练开发者在多模块协同、接口设计、部署运维和结果表达方面的能力。

第三，能够体现深度学习与工程实现的结合能力。深度学习本身偏向算法研究，但在实际应用中必须与接口服务、数据库、页面交互和部署方案结合，才能形成真正可交付的软件系统。通过本课题，不仅可以掌握卷积神经网络和迁移学习的基本思想，也可以理解如何把模型能力包装为可被用户直接使用的系统功能。

第四，具有较好的教学与展示价值。系统能够上传图片、展示识别结果、保存历史记录、查看后台统计，具备较好的交互性和可视化效果，适合在毕业答辩、课程展示和项目演示中使用。

### 1.3 国内外研究现状

在图像分类与目标检测研究方面，国内外已经形成了较为成熟的理论基础和应用体系。早期图像识别方法主要依赖人工设计的特征，如 SIFT、HOG、LBP 等，这些方法对特征设计者的经验依赖较强，在复杂场景中鲁棒性有限。随着 AlexNet 在 ImageNet 挑战赛中的突破，卷积神经网络开始成为图像识别领域的核心技术。随后 VGG、GoogLeNet、ResNet、DenseNet、EfficientNet 等模型不断推动图像分类性能提升。

在目标检测方向，R-CNN、Fast R-CNN、Faster R-CNN、Mask R-CNN 等两阶段检测方法在精度上取得了显著进展；YOLO 系列、SSD 系列等单阶段检测方法则在速度和工程部署方面具有优势。尤其在道路交通感知任务中，目标检测相比整图分类更适合处理多目标场景，因此广泛应用于自动驾驶和智能交通系统。

从工程系统角度看，当前很多研究不仅关注模型精度，还关注模型如何在 Web 服务、边缘设备和云平台中部署。例如使用 Flask、FastAPI、Django 等框架将深度学习模型包装为 API 服务，结合前端页面、数据库和容器化环境完成应用落地。对于本科毕业设计而言，这种“算法 + 系统 + 部署”的综合实现方式比单纯模型实验更能体现实际能力。

然而，已有研究也存在一些问题。首先，很多论文停留在数据集指标层面，没有形成可直接演示和使用的软件系统；其次，自动驾驶场景对数据和算力要求高，本科阶段往往难以完成从采集、标注到训练优化的全流程；再次，多技术栈组合虽然功能丰富，但开发成本高、调试周期长，容易影响最终交付质量。因此，本课题在吸收相关研究思路的基础上，采取以“工程可交付”为核心的实现策略。

### 1.4 本文主要研究内容

本文围绕一套基于深度学习的图片分类系统展开研究与实现，主要工作包括以下几个方面。

1. 对任务书和开题报告中的业务需求进行梳理，明确系统需要覆盖的七类道路目标和前后台功能。
2. 结合道路多目标场景特点，设计“目标检测主引擎 + 分类模型回退”的双层推理方案。
3. 采用 FastAPI、Jinja2、原生 JavaScript 和 SQLite 设计并实现前端页面、后台接口和数据存储结构。
4. 完成图片上传、推理、结果展示、历史记录、统计分析、健康检查等关键功能。
5. 在服务器环境中完成独立部署，构建 systemd 服务并验证运行状态。
6. 形成论文、摘要、答辩问答和进度落实说明等配套文档，支撑毕业设计整体交付。

### 1.5 论文结构安排

本文共分为七章。第一章介绍课题背景、意义、国内外现状及本文工作内容；第二章分析系统需求与总体方案；第三章介绍相关理论基础与关键技术；第四章给出系统总体设计；第五章说明系统具体实现过程；第六章对系统进行测试验证与结果分析；第七章总结全文并展望未来工作。

## 第二章 需求分析与总体方案

### 2.1 业务场景分析

根据任务书和开题报告，本课题聚焦自动驾驶道路目标分类。自动驾驶车辆在行驶过程中，需要感知道路环境中出现的多种交通参与者和交通设施，例如行人、车辆、交通标志和信号灯等。这些目标的存在直接影响驾驶安全，因此构建一个面向道路场景的图片识别系统，是一个合理且具有现实意义的课题切入点。

与普通图片管理系统不同，本系统的输入不是任意图片，而是更偏向交通道路场景图片；其输出也不是简单的文件管理结果，而是与道路目标相关的分类信息和可视化结果。因此，系统设计必须围绕“上传图片—执行识别—展示结果—保存记录—后台统计”这一业务链路展开。

### 2.2 功能需求分析

系统主要功能需求可分为用户端需求和管理端需求两个部分。

用户端需求包括：

1. 用户能够上传本地图片文件；
2. 系统能够接收图片并调用深度学习模型进行识别；
3. 页面能够展示主要预测类别和置信度；
4. 页面能够展示模型推理模式，便于说明当前是检测模型还是分类回退模型在工作；
5. 若检测到目标，系统能够生成带检测框的可视化图片；
6. 用户能够查看识别目标的明细信息；
7. 用户能够查看最近的历史识别记录。

管理端需求包括：

1. 管理员能够查看系统总识别次数；
2. 能够查看类别分布和最近识别时间；
3. 能够查看最近十条识别记录；
4. 可选支持管理员口令，体现基本安全控制。

系统辅助需求包括：

1. 提供健康检查接口，便于部署后巡检；
2. 保存识别记录到数据库，便于后续统计和演示；
3. 在服务器上以独立服务形式运行，不影响现有服务。

### 2.3 非功能需求分析

非功能需求主要包括以下内容。

第一，可交付性。系统必须在毕设周期内完成，不能只停留在设计文档或伪代码层面，而要能够真正启动、访问、上传图片并返回结果。

第二，可维护性。项目代码结构应尽量清晰，前端、后端、模型和数据存储模块边界明确，便于后续阅读和修改。

第三，可部署性。系统需要能够部署到指定服务器中，最好采用独立目录、独立端口和独立 systemd 服务，避免影响原有业务。

第四，可解释性。由于项目用于毕业设计答辩，系统需要具备较好的可展示性和可说明性，既要让功能看得见，也要能清楚解释技术路线和工程化折中。

第五，稳定性。考虑到演示环境可能缺少 GPU，系统应支持 CPU 环境运行，并在模型加载失败时给出明确错误信息。

### 2.4 可行性分析

#### 2.4.1 技术可行性

PyTorch 和 torchvision 提供了成熟的预训练模型，可直接用于图像分类和目标检测任务。FastAPI 具有轻量、开发效率高、接口文档友好等优点，适合快速搭建深度学习推理服务。SQLite 部署简单，无需额外数据库服务，适合中小型项目和毕业设计环境。因此，本系统在技术上是可行的。

#### 2.4.2 时间可行性

如果严格按开题中的小程序、Spring Boot、Vue3、MySQL 和模型训练全链路实施，开发量较大，调试成本高，不利于在有限时间内形成高质量可交付作品。采用 Python 单体方案并复用预训练模型，可以显著减少样板代码和环境联调工作，从而保证项目在时间上可行。

#### 2.4.3 资源可行性

本科毕设通常难以获得足够训练数据、高性能 GPU 和长时间调优条件。因此，采用预训练模型进行迁移应用，并将重点放在系统实现、模型接入和部署验证上，是一种符合现实资源条件的可行策略。

### 2.5 总体技术路线

综合需求分析和可行性分析，本文确定如下总体技术路线：

1. 使用 FastAPI 构建后端服务，负责页面渲染、接口处理、图片接收和模型调用；
2. 使用 MobileNetV2 预训练分类模型作为核心推理引擎；
3. 将 ImageNet Top-5 标签映射到道路七类目标，输出适合课题要求的结果；
4. 在没有明确映射结果时，将图片归类为未识别类别；
5. 使用 SQLite 保存识别记录和统计数据；
6. 使用 Jinja2 + HTML + CSS + JavaScript 实现用户端和管理端页面；
7. 使用 systemd 管理服务，完成远程部署与验证。

## 第三章 相关技术与理论基础

### 3.1 深度学习与卷积神经网络概述

深度学习是机器学习的重要分支，其核心思想是通过多层神经网络自动学习数据中的层次化特征表示。在图像任务中，卷积神经网络通过局部感受野、权值共享和池化机制，能够高效提取边缘、纹理、形状和语义等特征，显著优于传统人工特征方法。卷积神经网络通常由卷积层、激活函数、池化层和全连接层等组成，能够从大量样本中学习分类或检测规则。

对于图像分类任务，卷积神经网络会输出每个类别的概率分布，选取概率最高的类别作为预测结果。对于目标检测任务，网络不仅要识别图像中是什么目标，还要给出目标位置信息，因此难度更高，但表达能力也更强，更适合道路场景。

### 3.2 迁移学习思想

迁移学习是指将一个任务上学到的知识迁移到另一个相关任务中，以减少目标任务所需的数据和训练成本。ImageNet、COCO 等公开数据集上的预训练模型已经学习到大量通用视觉特征，在本科毕设中直接利用这些模型，能够显著缩短开发周期，提高起步效果。

本项目采用的 MobileNetV2 模型来自公开预训练权重，并通过 ONNX Runtime 在 CPU 环境中进行推理。虽然该模型不是针对七类道路目标专门训练的，但其已具备较强的通用视觉表达能力，通过类别映射和工程包装，可以快速形成可运行的道路识别系统。这体现了迁移学习与轻量化部署在工程场景中的应用价值。

### 3.3 MobileNetV2 分类模型

MobileNetV2 是一种面向移动端与轻量化场景设计的卷积神经网络，通过深度可分离卷积与倒残差结构在保持较高精度的同时显著降低参数量和计算量，非常适合资源受限环境中的部署推理。

本项目选用 MobileNetV2 作为部署模型，原因在于其模型体积较小、CPU 推理友好、ONNX 生态成熟，能够在指定服务器资源条件下稳定安装与运行。对于毕业设计交付版而言，这种轻量化方案比重量级检测模型更容易实现真实上线。

### 3.4 ONNX 与 ONNX Runtime 部署技术

ONNX 是开放的神经网络交换格式，可以在不同框架和运行时之间复用模型。ONNX Runtime 则是高性能推理引擎，支持 CPU 环境下的轻量部署，特别适合需要快速交付和跨环境运行的深度学习应用。

在本项目中，ONNX Runtime 负责加载和执行 MobileNetV2 模型。通过将预训练模型导出为 ONNX 格式，系统可以避开大型训练框架的复杂依赖，从而在普通 CPU 服务器上完成真实部署。这种设计保证了系统具备深度学习推理能力，同时显著降低了上线门槛。

### 3.5 FastAPI 技术特点

FastAPI 是基于 Python 的现代 Web 框架，具有开发效率高、类型提示友好、性能较好、异步支持良好等特点。它特别适合与机器学习和深度学习项目结合，因为模型调用通常使用 Python 生态，采用 FastAPI 可以减少语言切换成本。

在本系统中，FastAPI 不仅承担接口服务功能，也负责页面路由和静态资源挂载。通过统一的 Python 服务，可以同时管理上传、推理、数据库访问和页面渲染，大幅降低工程复杂度。

### 3.6 SQLite 数据存储特点

SQLite 是一种轻量级关系型数据库，数据库文件直接存储在本地磁盘，不需要单独部署数据库服务。其优点包括：零配置、便于迁移、适合中小规模数据以及对开发和演示环境非常友好。

本项目中的识别记录规模相对较小，主要用于历史留痕和统计展示，因此使用 SQLite 是合理的。虽然它不如 MySQL 适合高并发场景，但对于本科毕设交付版而言，SQLite 具有更高的实施效率和部署便利性。

## 第四章 系统设计

### 4.1 系统总体架构设计

本系统总体上可分为表示层、业务层、模型层和数据层四个部分。

1. 表示层：包括用户端页面和管理端页面，负责与用户交互，展示识别结果和统计信息；
2. 业务层：由 FastAPI 接口构成，负责处理上传请求、调用模型、组织返回数据和控制访问流程；
3. 模型层：包括目标检测主模型和分类回退模型，负责执行深度学习推理；
4. 数据层：采用 SQLite 存储预测记录，保存图片路径和识别结果。

系统运行流程为：用户上传图片后，后端先将文件保存到本地目录，再调用模型引擎执行推理；若主模型检测到目标，则返回映射后的主要类别和检测明细，并在图像上绘制标注框；随后系统将结果写入数据库，并在前端页面上展示；管理端则通过统计接口获取累计数据和最近记录。

### 4.2 模块划分设计

#### 4.2.1 页面展示模块

页面展示模块包括首页和后台页。首页用于上传图片、展示识别结果、显示七类目标介绍和历史记录。后台页用于查看统计信息和最近识别记录。通过两个页面的配合，可以较完整地展示系统的用户视角和管理员视角。

#### 4.2.2 推理服务模块

推理服务模块负责模型加载、目标检测、类别映射、回退分类和标注图生成。为了避免重复加载模型，系统在进程内采用缓存方式保留模型实例，首次请求时完成初始化，后续请求重复使用已加载模型。

#### 4.2.3 数据管理模块

数据管理模块负责创建数据库表、插入预测记录、读取历史记录和统计类别分布。数据库表中保存文件名、图片路径、预测类别、置信度、模型模式、原始 JSON 和创建时间等字段，为结果追溯和后台展示提供基础。

#### 4.2.4 部署与运维模块

部署与运维模块包括健康检查接口、远程部署脚本、systemd 服务配置和日志查看方式。这样做的目的，是让系统不仅能在本地运行，也能在服务器中长期稳定运行，体现项目的工程完整性。

### 4.3 前端交互设计

首页采用卡片式布局，顶部展示项目简介和系统能力，主体部分包含上传区、识别结果区、支持类别区和历史记录区。上传区引导用户选择图片，点击按钮后触发异步请求；识别结果区展示原图、主类别、置信度、模型模式、目标明细和标注图；历史记录区展示最近识别样本。

后台页采用统计卡片加记录列表的形式，便于快速查看系统总体运行状态。若配置了管理员密码，则请求后台接口时需通过请求头传递口令，实现简单安全控制。

### 4.4 接口设计

系统设计了四类核心接口。

1. `/api/health`：用于健康检查，返回服务状态和时间戳；
2. `/api/classify`：用于接收上传图片并返回识别结果；
3. `/api/history`：用于查询最近历史记录；
4. `/api/admin/stats`：用于查询统计信息和最近记录，必要时需要管理员口令。

接口采用 JSON 作为数据交换格式，便于前端异步调用和后续拓展。

### 4.5 数据库设计

数据库表 `predictions` 结构如下：

- id：主键，自增；
- filename：上传文件名；
- original_path：原始图片相对路径；
- annotated_path：标注图相对路径；
- predicted_class：系统内部类别键；
- predicted_label：中文类别名称；
- confidence：置信度；
- model_mode：模型模式；
- raw_json：原始检测结果 JSON；
- created_at：创建时间。

该设计既能满足前端展示，也为论文中“数据管理模块”的论述提供了依据。

### 4.6 类别映射设计

为了满足课题要求中的七类道路目标输出，系统建立了两个映射关系。

第一类映射是 COCO 到系统类别的映射。例如 `person` 映射为行人，`traffic light` 和 `stop sign` 映射为交通标志/信号灯，`bus`、`truck` 等直接映射到对应类别。

第二类映射是 ImageNet 到系统类别的回退映射。例如 `sports car`、`cab`、`jeep` 等归入汽车，`school bus` 归入公交车，`pickup`、`tow truck` 归入卡车。这种映射设计虽然不是严格的数据集再训练过程，但足以支撑毕业设计中的演示和分析需求。

### 4.7 安全与部署设计

为了避免影响服务器已有服务，系统部署时采用独立目录、独立端口和独立 systemd 服务。管理端统计接口可选择启用管理员口令，通过请求头访问，增强基本安全性。系统日志由 `journalctl` 查看，便于运维排查。

## 第五章 系统实现

### 5.1 项目目录实现

系统目录围绕职责划分组织。`app/` 下保存后端主程序、数据库逻辑、模型引擎、静态资源和模板文件；`data/` 存储上传图片、标注图片和数据库文件；`docs/` 保存论文与答辩文档；`scripts/` 保存部署和测试脚本。这种结构清晰、职责明确，便于展示和维护。

### 5.2 用户端页面实现

首页通过模板渲染项目名称和七类目标元信息。上传表单使用浏览器原生文件选择组件，结合 JavaScript 构造 FormData 发起异步请求。为了提升交互体验，系统在上传后立即显示原图预览，并在请求完成后动态渲染识别结果、模型模式和目标明细。

结果展示部分分为原图卡片和识别结果卡片。识别结果卡片中，主类别和置信度采用较明显的样式突出显示；若系统生成标注图，则在下方显示带框图像；否则隐藏该区域，避免界面冗余。

历史记录区域通过异步请求获取最近识别结果，并以卡片形式展示图片、类别、置信度、模型类型和时间，增强了系统的可视化效果和可追溯性。

### 5.3 管理端页面实现

后台管理端主要通过 `/api/admin/stats` 接口加载数据。页面上方提供管理员口令输入框，若系统设置了环境变量 `ADMIN_PASSWORD`，则前端会将口令通过 `X-Admin-Password` 请求头带入请求。接口成功后，页面渲染总识别次数、最近识别时间和类别分布信息，同时展示最近十条识别记录。

这种设计虽然简单，但足以体现后台管理功能，不仅满足任务书要求，也便于答辩时展示“系统不仅有用户功能，也具备后台统计管理能力”。

### 5.4 后端接口实现

后端主程序在启动时自动创建上传目录并初始化数据库。`/api/classify` 接口的主要实现步骤如下：

1. 接收上传文件；
2. 检查文件内容是否为空；
3. 以时间戳和随机串组成唯一文件名，保存到上传目录；
4. 调用模型引擎执行推理；
5. 将结果组织为统一结构，包括主要类别、置信度、模型模式、检测列表、原始图路径和标注图路径；
6. 将记录写入 SQLite；
7. 以 JSON 返回前端。

`/api/history` 接口从数据库读取最近记录，并补充可访问的图片 URL。`/api/admin/stats` 接口在通过权限校验后返回统计结果和最近记录。`/api/health` 则用于部署后巡检。

### 5.5 模型引擎实现

模型引擎是系统的核心模块。实现逻辑分为以下几个部分。

第一，模型加载。系统在首次调用时下载并加载 MobileNetV2 ONNX 模型与标签文件，并缓存到进程内全局变量中，避免每次请求重复初始化。

第二，图像预处理。对于输入图片，系统按 ImageNet 预处理流程执行缩放、中心裁剪、归一化与张量维度变换，为 ONNX 模型推理做好准备。

第三，可视化。若检测到有效目标，则在图片上绘制边界框和标签，并保存标注图路径，以供前端展示。

第四，类别映射。系统选取 Top-5 候选标签，将其中可映射到道路七类目标的标签作为最终结果；如果没有命中映射，则归为未知类别。

第五，异常处理。模型初始化或推理失败时，系统统一抛出引擎不可用异常，由接口层转换为友好的错误提示，避免前端直接暴露底层异常细节。

### 5.6 数据持久化实现

数据库操作模块主要实现三个功能：初始化数据表、插入识别记录和查询统计信息。初始化时使用 `CREATE TABLE IF NOT EXISTS` 保证幂等性；插入记录时保存完整识别元数据；统计时通过 SQL 聚合语句计算总次数和各类别出现次数。这些设计使系统在多次启动和多次演示过程中能够保持数据连续性。

### 5.7 健康检查与日志实现

健康检查接口会返回服务名称、状态和 UTC 时间戳，便于本地和远程脚本检测服务运行状况。部署后可通过 `curl` 或反向代理探测进行检查。服务运行日志可通过 `journalctl -u bishe-image-classification.service` 查看，便于诊断模型加载失败、依赖缺失或端口占用等问题。

### 5.8 服务器部署实现

为了满足毕业设计“形成成品并完成部署”的要求，系统设计了远程部署脚本。该脚本能够完成项目打包、上传、解压、创建虚拟环境、安装依赖、生成 systemd 服务、启动服务并执行健康检查。部署参数包括远程主机、部署目录、服务名称、端口和管理员密码等。

在部署策略上，系统选择使用独立目录 `/opt/bishe-image-classification`、独立端口 `19001` 和独立服务 `bishe-image-classification.service`，这样可避免影响服务器现有业务环境，符合任务要求中的隔离部署原则。

系统测试分为本地测试和服务器测试两个部分。本地测试环境为 Linux + Python 3，使用虚拟环境安装 FastAPI、uvicorn、torchvision 等依赖；服务器测试环境使用独立目录和 systemd 管理服务，确保上线后具备持续运行能力。

功能测试围绕核心流程展开。

向 `/api/health` 发送 GET 请求，系统能够返回包含 `status=ok` 的 JSON 数据，说明服务进程正常运行。

#### 6.2.2 图片上传测试

用户在首页上传图片后，系统能够成功接收文件并保存到 `data/uploads/` 目录中，说明上传流程正常。

#### 6.2.3 推理结果展示测试

上传包含车辆或道路目标的图片时，系统能够返回预测类别、置信度、模型模式、原图路径及标注图路径；前端页面能够正确渲染结果卡片，说明前后端联动正常。

#### 6.2.4 历史记录测试

完成多次识别后，访问历史接口，系统能够按时间倒序返回最近记录，并在首页历史区域中展示，说明数据库写入和查询逻辑正常。

#### 6.2.5 后台统计测试

访问后台统计接口，系统能够返回总识别次数、类别分布和最近十条记录。若配置管理员口令，使用正确口令可正常访问，错误口令则返回未授权信息，说明权限控制逻辑正常。

为提高验证效率，系统提供了 `scripts/smoke_test.sh` 脚本。该脚本依次执行健康检查、历史记录检查和后台统计检查，适合本地与远程环境快速验证。通过脚本化测试，能够证明系统不仅能在浏览器中访问，也能从运维角度完成基础接口巡检。

### 6.4 推理效果分析

由于本项目采用的是预训练模型映射方案，而不是基于专用道路七分类数据集重新训练，因此其优势与不足都较为明显。

优势方面：

1. 起步效果较好，能够较快识别常见交通目标；
2. 检测框可视化效果直观，便于结果分析与系统展示；
3. 不依赖长时间训练，适合本科毕设周期。

不足方面：

1. 类别映射方式存在一定信息损失；
2. 对罕见场景、复杂遮挡和特殊角度图片的识别能力有限；
3. 在 CPU 环境中推理速度难以达到任务书中提到的工业级实时指标。

尽管如此，考虑到本科毕业设计强调“问题分析、系统设计、工程实现和完整交付”，该方案仍然具有较高的合理性。它能够展示模型接入、结果可视化、数据库管理和服务器部署等完整链路，远优于只提交模型脚本或只做界面原型的方案。

### 6.5 工程化验证分析

从工程角度看，本系统通过以下方式证明其可交付性：

1. 具备完整前台页面和后台页面；
2. 具备真实深度学习推理能力；
3. 具备识别记录持久化能力；
4. 具备健康检查和脚本化测试能力；
5. 具备独立部署和服务托管能力；
6. 具备论文、摘要、答辩材料等文档支撑。

这说明本项目并非单点功能演示，而是一套从需求到部署较为完整的毕业设计成品。

### 6.6 与任务书要求的对照分析

任务书中强调七大类道路目标、深度学习模型、图片上传、分类结果展示、后台管理和论文写作等内容。本文实现的系统已覆盖这些关键要求。虽然在技术选型上对原始计划进行了调整，但这种调整是为了更稳妥地完成可运行交付，并未削弱课题本身的研究和实现价值。

### 6.7 存在的问题与风险分析

目前系统仍存在以下问题。

第一，模型并非专门基于 KITTI 或自建道路七分类数据训练，因此在严格学术指标上无法直接对标专用模型。

第二，系统采用 SQLite 和一体化页面方案，更偏向轻量演示，不适用于高并发生产环境。

第三，系统在第一次推理时需要下载预训练模型权重，对网络环境有一定依赖。

第四，若部署服务器缺少足够内存或算力，模型初始化速度可能偏慢。

这些问题在论文和答辩中应主动说明，并强调本项目的定位是“本科毕业设计可交付版”，重点体现工程闭环与系统完整性。

## 第七章 总结与展望

### 7.1 工作总结

本文围绕《基于深度学习的图片分类系统》这一本科毕业设计课题，完成了一套可运行、可部署、可展示的道路目标识别系统。系统以自动驾驶道路场景为应用背景，针对七类目标构建了完整的识别流程，完成了用户端上传、深度学习推理、结果可视化、历史记录、后台统计、数据存储和服务器部署等关键模块。

在技术实现上，本文采用 ONNX Runtime + MobileNetV2 形成轻量化深度学习推理方案；在系统实现上，采用 FastAPI、Jinja2、JavaScript 和 SQLite 构建轻量而完整的 Web 系统；在工程落地上，完成了独立部署、systemd 托管和基础脚本化验证。整体来看，项目已经较好地实现了任务书和开题报告中的核心目标。

### 7.2 项目特色

本项目的主要特色体现在以下几个方面。

1. 面向道路交通场景，而非泛化的图片分类任务，应用背景明确；
2. 采用目标检测与分类回退结合的策略，提高系统稳定性；
3. 将深度学习模型真正集成到 Web 系统中，形成前后台一体化展示效果；
4. 从算法、接口、数据库到部署形成完整交付闭环，更符合毕业设计要求。

### 7.3 不足之处

尽管系统已经可以运行和部署，但仍存在进一步提升空间。首先，模型未经过专门数据集微调，识别效果受限于通用预训练权重；其次，系统在 CPU 环境下推理速度有限，尚不能满足车载实时系统要求；再次，前后端和数据库方案采用了以可交付为优先的简化实现，与工业级系统架构仍有差距。

### 7.4 后续优化方向

未来可从以下方向继续优化。

1. 采集或整理道路场景专用数据集，对七类目标进行有监督微调，提高精度；
2. 使用更轻量或更高效的检测模型，如 YOLOv8、PP-YOLOE 或 MobileNet 变体；
3. 使用 ONNX、TensorRT、OpenVINO 等方案提升部署效率；
4. 将 SQLite 升级为 MySQL，将页面升级为 Vue 前后端分离架构；
5. 增加批量识别、结果导出、用户鉴权和日志审计等工程功能；
6. 在论文后续研究中探索多模态感知、自监督学习和边缘推理优化等方向。

综上所述，本文完成的系统虽然仍属于毕业设计层面的工程实现，但已经能够较完整地体现一个深度学习图片分类系统从需求分析、方案设计、模型接入、功能实现到部署验证的全过程，达到了本科毕业设计成品交付的目标。
